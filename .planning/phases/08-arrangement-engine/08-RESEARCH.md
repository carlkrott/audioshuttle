# Phase 8: Arrangement Engine - Research

**Researched:** 2026-05-13
**Domain:** MIDI arrangement, song structure generation, timeline-aware DAW control
**Confidence:** HIGH

## Summary

The current `generate_project` creates markers correctly but generates a single MIDI file per instrument spanning `total_bars` with the SAME repeating pattern for every section. The MIDI is placed at bar 0 and runs linearly — there's no section-aware variation. Drums sound the same in the intro as the chorus. Bass plays root-fifth for 60+ bars. Melody just walks up the scale monotonically. No rests, no dynamics, no variation between sections.

The fundamental architecture problem: each instrument gets ONE pattern generated by `_generate_instrument_pattern()` that repeats identically for the entire song. There's no concept of "this section is a chorus, so make it louder/more active" or "this is an intro, so keep it sparse."

The fix requires a complete rewrite of the MIDI generation layer to be **section-aware** and **arrangement-aware**, plus marker-aware placement so MIDI items land at the correct timeline positions aligned with the markers.

**Primary recommendation:** Replace the single-pattern-per-instrument approach with a section-aware arrangement engine that generates different MIDI content per section type, places items at correct marker-aligned timeline positions, and uses E2B for musical quality assessment.

## Problems with Current Implementation

### Problem 1: Monolithic Pattern Generation
```python
# Current: ONE pattern for ENTIRE song
gen_bars = max(4, min(total_bars, 32))
self._generate_instrument_pattern(midi, mtrack, role, scale_notes, total_bars, bpm)
```
This creates a single MIDI file with identical content across all sections. The "drums" role generates kick-on-1-3/snare-on-2-4 for every single bar regardless of whether it's an intro or chorus.

### Problem 2: No Section-Aware Variation
`_generate_instrument_pattern()` has zero knowledge of sections. It just loops `gen_bars` times doing the same thing. There's no concept of:
- Intro: sparse, building
- Verse: moderate energy, establishing groove
- Chorus: full energy, maximum activity
- Bridge: different texture, contrast
- Outro: winding down

### Problem 3: MIDI Placement at Bar 0 Only
Each instrument's MIDI file starts at beat 0. The MIDI import via Lua watcher inserts at the edit cursor position. But `generate_project` never moves the cursor — it just dumps the file and Reaper places it at the start of the track.

### Problem 4: No Track Awareness
`generate_project` creates new tracks blind — it doesn't check what already exists. Running it twice creates duplicate tracks.

### Problem 5: No E2B Musical Intelligence
The E2B model is used for NLU (command translation) but never for musical assessment. It could validate arrangements, suggest variations, assess whether the pattern is musically coherent.

## Architecture Patterns

### Recommended: Section-Aware Arrangement Engine

```python
# Each section type has a DENSITY profile
SECTION_PROFILES = {
    "intro":   {"density": 0.3, "velocity_range": (50, 80),  "active_roles": {"drums", "bass", "keys"}},
    "verse":   {"density": 0.6, "velocity_range": (60, 100), "active_roles": {"drums", "bass", "keys", "melody"}},
    "chorus":  {"density": 1.0, "velocity_range": (80, 127), "active_roles": ALL},
    "bridge":  {"density": 0.5, "velocity_range": (50, 90),  "active_roles": {"keys", "strings", "arp"}},
    "outro":   {"density": 0.4, "velocity_range": (40, 80),  "active_roles": {"drums", "bass", "keys"}},
}
```

### Pattern: Per-Section MIDI Generation

Instead of one big MIDI per instrument:
1. For each section, for each instrument, generate section-appropriate MIDI
2. Place it at the correct beat offset: `section_start_bar * 4`
3. Duration = `section_bars * 4` beats
4. Velocity/density scaled by section profile

### Pattern: Multi-Item Timeline Placement

For proper Reaper arrangement:
1. Generate one MIDI file per instrument PER SECTION
2. Import each at the correct track + timeline position
3. Result: discrete MIDI items on the timeline, aligned with markers

OR (simpler, fewer files):
1. Generate one MIDI file per instrument containing ALL sections
2. Use beat offsets to place notes at correct section positions
3. Import once per track
4. Reaper shows it as one continuous MIDI item (less flexible but simpler)

**Recommendation:** Start with the simpler approach (one MIDI per instrument with beat offsets), but use MIDI markers/meta events to denote section boundaries. Can upgrade to per-section items later.

### Pattern: DAW State Enforcement

Before any arrangement operation:
1. Always call `refresh_state()` or equivalent
2. Check existing tracks, markers, tempo
3. Only create what doesn't exist
4. Place MIDI relative to existing content

## Common Pitfalls

### Pitfall 1: MIDI Overlap on Re-import
**What goes wrong:** If tracks already have MIDI from a previous `generate_project`, re-running appends new MIDI at bar 0, creating overlap.
**Why it happens:** Lua watcher just does `InsertMedia` at edit cursor, doesn't clear existing items.
**How to avoid:** Either clear track items first (new Lua trigger for "clear track N items"), or make `generate_project` idempotent by checking state.
**Warning signs:** Notes sounding doubled, wrong durations.

### Pitfall 2: Section Name Matching
**What goes wrong:** User says "Verse" but section profile lookup expects exact "verse".
**Why it happens:** Case sensitivity, spaces, numbering ("Verse 1" vs "verse").
**How to avoid:** Normalize section names: lowercase, strip numbers, then lookup.
**Warning signs:** All sections get "generic" profile instead of section-specific.

### Pitfall 3: Beat Offset Calculation in Minor vs Major
**What goes wrong:** Minor scale has different intervals than major, but pattern generation assumes major.
**Why it happens:** `_generate_instrument_pattern` uses `scale_notes` correctly, but section profiles might assume major harmony.
**How to avoid:** Use `_scale_notes(key, scale)` which already handles this. Chord generation should use scale-appropriate degrees (i-iv-v in minor, not I-IV-V).

### Pitfall 4: E2B Model Not Running
**What goes wrong:** Musical assessment calls fail silently.
**Why it happens:** E2B port 8093 is currently DOWN.
**How to avoid:** Make arrangement work WITHOUT E2B assessment (deterministic fallback), with E2B as optional enhancement. Check model availability before calling.

## Code Examples

### Section-Aware Pattern Generation (Replacement for `_generate_instrument_pattern`)

```python
def _generate_arrangement(
    self,
    midi: "MIDIFile",
    mtrack: int,
    role: str,
    scale_notes: list[int],
    sections: list[dict],  # [{"name": "Verse 1", "bars": 16}, ...]
    bpm: int,
) -> None:
    """Generate section-aware MIDI arrangement for one instrument."""
    
    bar_offset = 0
    for section in sections:
        name = str(section["name"]).lower().split()[0]  # "verse 1" → "verse"
        bars = int(section["bars"])
        profile = SECTION_PROFILES.get(name, SECTION_PROFILES["verse"])
        
        # Check if this role is active in this section
        if role not in profile["active_roles"]:
            bar_offset += bars
            continue  # Rest in this section
        
        density = profile["density"]
        vel_lo, vel_hi = profile["velocity_range"]
        
        # Generate section-appropriate pattern
        self._generate_section_pattern(
            midi, mtrack, role, scale_notes,
            start_bar=bar_offset, bars=bars,
            density=density, vel_range=(vel_lo, vel_hi),
            section_type=name, bpm=bpm,
        )
        bar_offset += bars
```

```python
def _generate_section_pattern(self, midi, mtrack, role, scale_notes,
                               start_bar, bars, density, vel_range, section_type, bpm):
    ch = 9 if role in ("drums", "drum", "beat", "kick", "rhythm") else 0
    
    if role in ("drums", "drum", "beat", "kick", "rhythm"):
        for bar in range(bars):
            off = (start_bar + bar) * 4
            vel_base = random.randint(vel_range[0], vel_range[1])
            
            # Kick - density controls how many kick hits
            if random.random() < density:
                midi.addNote(mtrack, ch, 36, off, 1, vel_base)
            if random.random() < density * 0.7:
                midi.addNote(mtrack, ch, 36, off + 2, 1, int(vel_base * 0.9))
            
            # Snare
            if random.random() < density:
                midi.addNote(mtrack, ch, 38, off + 1, 1, vel_base)
            if random.random() < density * 0.5:
                midi.addNote(mtrack, ch, 38, off + 3, 1, int(vel_base * 0.8))
            
            # Hi-hat
            for b in range(int(8 * density)):
                hat_time = off + b * (4.0 / max(1, int(8 * density)))
                midi.addNote(mtrack, ch, 42, hat_time, 0.5, int(vel_base * 0.6))
            
            # Ghost notes in chorus
            if section_type == "chorus" and random.random() < 0.3:
                midi.addNote(mtrack, ch, 38, off + 1.5, 0.5, int(vel_base * 0.4))
```

### Deterministic (No Random) Pattern with Section Variation

```python
def _drums_for_section(self, midi, mtrack, start_bar, bars, density, vel_range, section_type):
    """Deterministic drum pattern that varies by section."""
    ch = 9
    vel_mid = (vel_range[0] + vel_range[1]) // 2
    
    for bar in range(bars):
        off = (start_bar + bar) * 4
        
        # Always: kick on 1, snare on 2 and 4
        midi.addNote(mtrack, ch, 36, off, 1, vel_mid)
        midi.addNote(mtrack, ch, 38, off + 1, 1, vel_mid)
        midi.addNote(mtrack, ch, 38, off + 3, 1, int(vel_mid * 0.9))
        
        # Density controls hi-hat subdivision
        if density >= 0.8:  # Chorus: 16th notes
            for b in range(16):
                midi.addNote(mtrack, ch, 42, off + b * 0.25, 0.25, int(vel_mid * 0.5))
        elif density >= 0.5:  # Verse: 8th notes
            for b in range(8):
                midi.addNote(mtrack, ch, 42, off + b * 0.5, 0.5, int(vel_mid * 0.5))
        else:  # Intro/Outro: quarter notes only
            for b in range(4):
                midi.addNote(mtrack, ch, 42, off + b, 1, int(vel_mid * 0.4))
        
        # Chorus: extra kick on beat 3 ("four on the floor")
        if section_type == "chorus":
            midi.addNote(mtrack, ch, 36, off + 2, 1, int(vel_mid * 0.95))
```

### E2B Musical Assessment Call

```python
async def _assess_arrangement(self, key, scale, bpm, sections, instruments):
    """Ask E2B model to evaluate the musical arrangement."""
    prompt = (
        f"Rate this song arrangement from 1-10 for musical quality.\n"
        f"Key: {key} {scale}, BPM: {bpm}\n"
        f"Sections: {json.dumps(sections)}\n"
        f"Instruments: {instruments}\n\n"
        f"Respond with ONLY a JSON object: "
        f'{{"rating": N, "suggestions": ["suggestion1", ...], "energy_flow": "description"}}'
    )
    # Call E2B via model_server
    result = await self._model_server.chat(messages=[{"role": "user", "content": prompt}])
    return result
```

## Open Questions

1. **Randomness vs Determinism:** Should patterns use `random` for variation, or be fully deterministic (seed-based)?
   - **Recommendation:** Use `random.seed(hash(role + section_name + key))` for reproducible but varied patterns. Allow user to specify a seed.

2. **Per-section MIDI items vs single item:** Should each section be a separate MIDI item on the Reaper timeline, or one continuous item?
   - **Recommendation:** Start with one continuous item per instrument (simpler). The Lua watcher already handles single-file import. Per-section items require multiple imports per track which is fragile.

3. **E2B assessment integration:** Should assessment happen before or after generation?
   - **Recommendation:** AFTER generation, as validation/feedback. Don't gate generation on E2B availability. Generate deterministically, then ask E2B "is this good?" optionally.

4. **Clearing existing content:** Should `generate_project` clear existing MIDI on tracks before importing new?
   - **Recommendation:** Yes, add a "clear track items" trigger to the Lua watcher. New behavior: before import, send clear command for the target track.
